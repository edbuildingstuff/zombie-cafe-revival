# 2026-04-11 — Inheriting a reverse engineering project, and committing to a Godot rewrite

**Author:** Edward Yang ([@edbuildingstuff](https://github.com/edbuildingstuff))

This is the first entry in what I intend to be an ongoing devlog for the *Zombie Cafe Revival* project. The purpose of these entries is twofold: first, to keep an honest record of how the rewrite unfolds so I can reason about decisions later without having to re-derive them from commit messages, and second, to serve as raw material for longer-form write-ups I want to publish as the work progresses. Nothing here is polished. It's a journal.

## What I'm inheriting

The repo I'm starting from is Airyz's reverse engineering of *Zombie Cafe*, a 2011 Capcom mobile game whose online services were shut down years ago. If you want the technical primer, read [Airyz's own write-up](https://airyz.xyz/p/zombie-cafe-revival/) — it's the best description of the project's foundations and I'm not going to try to duplicate it here. What I'll record instead is what the repo actually looks like when you clone it fresh, because that's the thing I wish someone had handed me in one document.

At a high level, it's four things glued together:

1. **An extracted Android APK** — `src/smali/` has the 347 decompiled Dalvik classes (mostly obfuscated into single-letter filenames), `src/assets/` has eleven thousand game files, and `src/lib/armeabi/libZombieCafeAndroid.so` is the original ARMv7 native binary carrying the entire game engine. Nothing in the repo actually re-implements that binary; it just sits there as the runtime.

2. **A C++ runtime patcher** under `src/lib/cpp/`, called `LibZombieCafeExtension`. This is the most clever part of the existing work. It's a tiny shared library that gets loaded alongside the game's `.so` at startup, then walks to the game's base address and rewrites about fifteen byte ranges in memory using `mprotect` + memcpy. Those patches fix a texture destructor crash, redirect the game's hardcoded server URLs to `zc.airyz.xyz`, swap the "money buy" button to "toxin buy," and patch the version string. The game thinks it's still running the original 2011 code; it isn't quite.

3. **A Go workspace under `tool/`** with five modules: `build_tool` (orchestrates the APK rebuild), `file_types` (defines every binary format the game uses — save games, cafes, characters, food, furniture, animations), `cctpacker` (encodes and decodes the custom `CCTX` texture format), `resource_manager` (does JSON ↔ binary round-tripping so game data is human-editable), and `server` (a Cloudflare Workers backend that emulates Capcom's retired `/v1/zca/*` endpoints).

4. **No tests.** No CI. The whole pipeline is manual: `go run`, `cmake`, `apktool`, `jarsigner`, `adb install`.

The existing build works. It produces a playable APK that installs on a current Android phone, talks to Airyz's Cloudflare Workers backend, and runs. That by itself is an impressive amount of work, and none of what I'm about to write should be read as diminishing it. What Airyz shipped is the hard part.

## Why I'm not just extending this

I spent the morning reading through the repo with fresh eyes, and the thing that stood out to me is how much of this project's future is constrained by a single design decision: the runtime patcher treats `libZombieCafeAndroid.so` as a black box to be poked at, not as a binary to be replaced. That's a pragmatic choice — patching fifteen byte ranges is a lot cheaper than rewriting two megabytes of compiled C++ — but it has follow-on costs:

- The whole thing only runs on 32-bit ARM Android, because that's what the original `.so` was compiled for. No iOS, no desktop, no web, no ARM64 without recompilation work that nobody has the symbols for.
- The texture destructor patch is a NOP, not a fix. The game leaks a small amount of memory every time a texture unloads. That's fine in practice because a phone reboot clears it, but it'll never not be a NOP as long as the `.so` is the source of truth.
- The engine has a hardcoded limit of 2 character sheets, and the full Japanese version ships 5. You can't load the JP characters without patching the engine itself, which means writing more byte-range surgery against a binary nobody has the debug symbols for.
- The animation format has been decoded but isn't re-packed by the Go tooling yet, because until the engine can actually consume the re-packed output, there's no point finishing that code.

Every one of those limits is a direct consequence of "the original `.so` is still running the game." If the `.so` stops running the game, they all disappear at once.

## The three paths I considered

There are basically three ways to lift those constraints. I went through each of them honestly before committing.

**Path A** is to wrap the existing `.so` in an ARM user-mode emulator — something like `unidbg` or QEMU user-mode — and stub out the JNI calls and GL ES 1.0 rendering. This is the fastest path to "runs on Windows." It's also the path where nothing fundamental changes: the memory leak is still there, the two-character-sheet ceiling is still there, you still don't own the code, and the emulator becomes a permanent maintenance dependency.

**Path B** is to statically decompile the `.so` in Ghidra and port the result to portable C++. The offsets already labelled in `ZombieCafeExtension.cpp` give you a head start on the symbol table. In the best case you end up with a bit-perfect recreation that runs anywhere. In the realistic case you spend six to twelve months producing a codebase that only reverse engineers can maintain, and every new feature fights the decompiled logic instead of building on it. Bit-perfect fidelity is the main reward, but this project has already deliberately diverged from the original (the toxin buy swap, the cafe-size upgrade removal, the server-side 90% save-drop throttle), so fidelity isn't something I'm actually optimizing for.

**Path C** is to rewrite the client in Godot 4, keep the entire Go asset pipeline and the Cloudflare Workers server, and treat the original `.so` as reference documentation rather than runtime code. Godot handles 2D sprite games of this scale effortlessly, exports to every platform I care about from a single codebase, and — this is the thing that tipped me over — the existing Go tooling was *already* written to pack and unpack game data, which is exactly what a Godot asset pipeline needs. The code I'd have to throw away is the Android shell and the runtime patcher. Everything else either stays as-is or gets a small new entry point.

I picked Path C. The full reasoning is in [`docs/rewrite-plan.md`](../rewrite-plan.md); the short version is that Godot wins on maintainability, wins on flexibility, wins on platform reach, and runtime performance is a non-issue because Zombie Cafe's peak entity count is trivially small by 2026 standards. The only axis where Path B is clearly better is bit-perfect behavioral fidelity, and I'm not optimizing for that.

## What's in the plan

The rewrite is phased, and the first phase is the one I expect to be least fun and most important: before writing any Godot code, I'm writing tests for the Go `file_types` package. Round-trip every `.bin.mid` file in `src/assets/data/` through `Deserialize → Serialize` and assert byte equality. The reason this goes first is that the Go binary format code is going to be the bridge between old saves and the Godot client, and if I start debugging Godot issues without being able to trust the bridge, every bug turns into a two-suspect investigation. I'd rather spend a day writing round-trip tests than a week chasing phantoms.

After that, Phase 1 is a new `-target godot` flag on `build_tool` that emits a Godot-friendly asset tree (PNGs, JSON, OGG, TTF) instead of an APK-friendly one. Phase 2 is the Godot project scaffold — one scene, a static cafe tile, a character sprite, and CI. Phase 3 is the save-load round-trip contract test between the Godot client and the Go tooling. Phase 4 is the tick loop and the bulk of the reverse engineering work. Phase 5 is the online features and the server cleanup. Phase 6 is the platform exports. Phase 7 is retiring the legacy Android build.

The `.so` stays in the repo throughout as an oracle: when I'm not sure how a specific behavior is supposed to work, I'll open Ghidra and look, using the offsets in `ZombieCafeExtension.cpp` as anchors. But nothing I write after today will depend on it at runtime.

## What I'm going to track in these devlogs

Going forward, each entry will cover: what I worked on, what I learned about the game's internals, any decision I'm making that I think will still matter in three months, and any behavior I reverse-engineered out of the binary. The last category is important because it's the part that's hardest to recover from a commit history — "I figured out that the customer spawn timer is stored as a little-endian float at offset 0x2c" is the kind of thing that belongs in a journal, not a commit message.

The entries are also written in roughly publishable form. Not polished — I'm not editing for blog output as I write — but structured so that a handful of them strung together will make a reasonable article. The decision-to-use-Godot story in this entry is probably the first one.
